## [التاريخ: 2026-07-19 — تغيير آلية التسمية: اسم الملف = العنوان المعروض]

### ما تم إنجازه
- عدّلت `buildSubjectPackage` بـ `githubPublisher.js` بطلب صريح من المدير: **استبدلت الترقيم التلقائي `lecture-01.pdf`, `lecture-02.pdf`...** باشتقاق مباشر: اسم الملف الفعلي = نفس العنوان اللي يكتبه الآدمن للعرض بقسم المادة (`title`) + `.pdf` مضمونة دائماً بالنهاية.
- أضفت `sanitizeFileTitle(title)`: تشيل امتداد `.pdf` لو الآدمن كتبه بنفسه (تفادي `.pdf.pdf`)، تشيل رموز غير صالحة بأسماء الملفات/الروابط (`\ / : * ? " < > |` + `؟ ،` العربية)، وتحوّل المسافات لـ `_` (نفس نمط مثال المدير `دارات_نظري_1`).
- أضفت معالجة تصادم الأسماء (`uniqueFileName`): لو نفس الاسم النهائي مستخدَم مسبقاً (إما بنفس دفعة الرفع أو بمحاضرات موجودة سلفاً بالمادة)، يُضاف لاحقة `_2`, `_3`... تلقائياً بدل الكتابة فوق ملف آخر بالغلط.
- حذفت الترقيم التسلسلي القديم بالكامل (`globalCounter`/`usedNumbers`) ودالة `pad2` (صارت بلا استخدام).

### مشكلات واجهتها
- أول تنفيذ كان يشيل رموز ASCII الممنوعة (`? " ...`) بس فاتني رمز الاستفهام العربي `؟` (يختلف عن `?` الإنجليزي بالـ code point) — لقطتها بنفسي أثناء الاختبار (سيناريو "رموز غير صالحة") وصلّحتها فوراً قبل التسليم.

### قرارات اتخذتها (ولماذا)
- التصادم يُحل بلاحقة رقمية على الاسم نفسه (`اسم_2.pdf`) بدل رفض الرفع أو الكتابة فوق ملف قديم — يحافظ على استمرارية العمل بلوحة التحكم بدون إيقاف الآدمن لحل تعارضات يدوياً، وبنفس الوقت ما يفقد أي ملف سابق.
- لو العنوان فاضي كلياً: fallback لاسم الملف الأصلي المرفوع من جهاز الآدمن (بدل نص عام مثل "ملف")، لأنه غالباً أوضح مرجعاً من نص جاهز.

### ملاحظات للمدير
- تحققت بستة سيناريوهات (عنوان بدون امتداد، عنوان فيه `.pdf` مسبقاً، رموز غير صالحة عربي/إنجليزي، تكرار بنفس الدفعة، تكرار مع محاضرة موجودة مسبقاً، وبلا عنوان إطلاقاً) — كلها بالنتيجة المتوقعة. و`npm run build` نجح كاملاً (64 وحدة).
- ⚠️ تنبيه استخدام: أي تغيير لاحق بعنوان محاضرة موجودة (تعديل `title` فقط، بدون رفع ملف جديد) **لن يُعيد تسمية الملف الفعلي على GitHub تلقائياً** — لأن `buildSubjectPackage` يبني اسم ملف جديد فقط للملفات المرفوعة حديثاً بمصفوفة `files`؛ العناصر بـ `existingLectures` تُنسخ كما هي بلا تغيير على `file`. لو المطلوب لاحقاً "غيّر اسم العرض ويُعاد تسمية الملف تلقائياً بالريبو"، هذي ميزة إضافية تحتاج تصميماً منفصلاً (rename على GitHub API، لا مجرد إعادة رفع).

## [التاريخ: 2026-07-19 — الدفعة 4: توكن لوحة التحكم + نشر بلا مراجعة يدوية]

### ما تم إنجازه
- ملف جديد `lib/adminAuth.js` (عقد لعضو 3): `validateToken(token, {owner, repo})` يتحقق فعلياً عبر GitHub API (401→توكن غير صالح، صلاحية push:false→بلا كتابة، ok→صالح)، و`storeToken`/`getStoredToken`/`clearToken` عبر `sessionStorage` فقط (يُمسح بإغلاق التبويب، أفضل أمنياً من localStorage لتوكن بصلاحية كتابة).
- صحّحت `DEFAULT_REPO` إلى `QPU_assistant_404` (بدل `assistant404-v2`) لتطابق الريبو المنشور فعلياً بالرابط المذكور بالخطة — ونقلت `DEFAULT_OWNER`/`DEFAULT_REPO` كمصدر وحيد بـ `adminAuth.js` يستوردهما `PublishPanel.jsx` بدل تكرارهما.
- `PublishPanel.jsx`: يقرأ `getStoredToken()` أولاً؛ لو موجود يتخطى حقل إدخال التوكن كلياً (رسالة "مستخدَم توكن الدخول المحفوظ" + زر "تغيير" اختياري لإظهار الحقل يدوياً عند الحاجة).
- `githubPublisher.js` — `publishToGitHub`: أضفت خطوة دمج تلقائي بعد فتح الـ PR (`PUT .../pulls/{number}/merge`, `merge_method: squash`)، مفعّلة افتراضياً (`autoMerge = true`). أضفت checkbox بـ `PublishPanel.jsx` "لا تدمج تلقائياً، اترك PR للمراجعة" (افتراضياً غير مفعّل) يمرر `autoMerge: false`.
- تعامل مع فشل الدمج بتدرّج لا بفشل كامل: لو الملفات رُفعت والـ PR فُتح لكن الدمج فشل (تعارض/حماية فرع)، العملية لا ترمي استثناء — ترجع `{ prUrl, merged:false, mergeError }` بدل فقدان كل التقدّم المُنجز فعلياً.

### مشكلات واجهتها
- لا شيء يُذكر. صياغة العقد بالخطة كانت دقيقة بما يكفي للتنفيذ المباشر.

### قرارات اتخذتها (ولماذا)
- عند فشل الدمج التلقائي فقط (لا فشل الرفع/فتح PR)، أرجع نتيجة جزئية ناجحة بدل رمي خطأ — لأن رمي خطأ هنا يُضيّع على المستخدم معرفة أن كل الملفات فعلاً وصلت وفُتح PR جاهز، ويضطره يعيد كل العملية من الصفر بلا داعٍ.
- استخدمت `merge_method: "squash"` لتاريخ commits نظيف بـ main (كل نشر = commit واحد)، بلا طلب صريح بالخطة لنوع الدمج — قرار تنفيذي بسيط، قابل للتغيير لو الإدارة تفضّل غيره.
- نقلت `DEFAULT_OWNER`/`DEFAULT_REPO` من `PublishPanel.jsx` إلى `adminAuth.js` (كلاهما ملفي) كمصدر وحيد، لأن `validateToken` تحتاجهما أيضاً ولا داعي لتكرارهما بملفين.

### هل احتجت تعديل ملف خارج نطاق ملكيتك؟ (يُفترض دائماً "لا" بهذا التصميم)

### ملاحظات للمدير
- `validateToken` تفترض أن استجابة GitHub تتضمّن `permissions.push` — هذا يظهر فقط إذا كان الـ token مصرَّح Repository permissions صراحة أو كان صاحبه Collaborator؛ لو رجع `permissions` غير موجود إطلاقاً (بعض إعدادات fine-grained tokens) سيُعتبر التوكن صالحاً طالما الوصول للريبو نجح — قد يحتاج عضو 6 اختباراً فعلياً بتوكن حقيقي محدود الصلاحيات للتأكد.

## [التاريخ: 2026-07-19 — إصلاح ما بعد المراجعة (الجولة 2، §2.2)]

### ما تم إنجازه
- أصلحت خللاً حقيقياً بـ `buildSubjectPackage` أكّدته مراجعتا عضو 6 والمدير: تعديل محتوى دكتور **نشِط حالياً** ضمن مادة متعددة الدكاترة، بدون تمرير `setActive` صراحة، كان يُصفِّر `active` إلى `false` بسبب `{...v, ...newVariant}` التي تكتب فوق الحالة السابقة بقيمة `shouldBeActive` المحسوبة (والتي كانت `false` دائماً في هذي الحالة بالذات).
- الإصلاح: الآن لو `setActive` غير مُمرَّرة صراحة (`typeof ... !== "boolean"`) ولدكتور موجود مسبقاً (`idx >= 0`) → تُحفَظ حالة `active` السابقة كما هي بدل افتراض `false`. لدكتور جديد كلياً (`idx === -1`) يبقى السلوك القديم: `active = true` فقط لو كان أول دكتور إطلاقاً بالمادة.
- الإصلاح معزول بالكامل داخل `githubPublisher.js` — لم ألمس `SubjectForm.jsx` ولا `AdminSectionsManager.jsx` (عضو 3) كما طلبت الإدارة.
- حدّثت التعليق التوثيقي أعلى الملف لتوضيح السلوك الجديد لـ `setActive`.
- تحقّق: كتبت اختباراً مستقلاً يحاكي بالضبط سيناريو التقرير (تعديل محتوى دكتور نشِط بدون `setActive`) → PASS. كما شغّلت `npm run build` كاملاً على المشروع بعد الإصلاح → نجح (63 وحدة، مطابق لتحقق عضو 6).

### مشكلات واجهتها
- لا شيء يُذكر — الخلل كان واضحاً ومحدد السطر بدقة من تقرير المراجعة.

### قرارات اتخذتها (ولماذا)
- اخترت "حفظ الحالة السابقة" بدل "دائماً true" أو "دائماً false" كافتراضي عند عدم تمرير setActive، لأن هذا هو السلوك الوحيد غير المفاجئ: تعديل محتوى دكتور يجب ألا يغيّر من هو الدكتور النشِط ضمنياً؛ تغيير من هو النشِط يجب أن يبقى فعلاً صريحاً (setActive) لا أثراً جانبياً لحفظ محتوى.
